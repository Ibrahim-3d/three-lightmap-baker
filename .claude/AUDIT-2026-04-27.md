## 1. Feature Inventory — What the Code Actually Has

### Atlas / UV Unwrap

**Implemented:**
- `src/lib/atlas/generateAtlas.ts:37` — `generateAtlas(meshes)` packs all meshes
  into a single shared [0,1]² atlas. Module-scoped `UVUnwrapper`; calls must be
  serial.
- `src/lib/atlas/generateAtlas.ts:71` — `generateAtlases(meshesByBin)` runs one
  serial xatlas pack per bin for multi-atlas pipelines.
- xatlas padding set to 16 and resolution to 4096 at:
  `src/lib/atlas/generateAtlas.ts:50-51`.
- `src/lib/atlas/renderAtlas.ts:101-131` — 25 jittered dilation offsets
  (±0px, ±1px, ±2px pattern) to bleed position/normal into seam-adjacent texels.
- `src/lib/atlas/renderAtlas.ts:133-138` — `AtlasResult` return type includes
  `positionTexture`, `normalTexture`, and a `dispose()` that releases both RTs.
- Normal transform: inverse-transpose of the model matrix upper-left 3×3 at
  `renderAtlas.ts:68`, correcting the pre-S8 `mat3(modelMatrix)` skew bug.

**Multi-atlas bin-packing:**
- `src/lib/utils/Packing.ts:117` — `binPackMeshes(meshes, opts)` greedy
  first-fit-decreasing packer. Inputs: `atlasResolution`, `texelsPerMeter`,
  `perMeshScale` (per-mesh density multiplier keyed by `mesh.uuid`), `fillRatio`.
- `src/lib/utils/Packing.ts:35` — `computeMeshSurfaceArea(mesh)` world-space
  triangle area sum.
- Auto-spawning of new atlas bins when density target overflows capacity:
  `src/lib/utils/Packing.ts:145-166`. Single mesh too large is clamped + warned.

### BVH Build + Per-Triangle Material Textures

- `src/lib/utils/GeometryUtils.ts:26` — `mergeGeometry(meshes)` adds a
  per-vertex `meshIndex` Float32Array attribute (fixes the S5 BVH-reorder bug)
  before `mergeGeometries`.
- `src/lib/utils/GeometryUtils.ts:124` — `extractPerTriangleMaterials(merged, meshes)`
  walks the post-BVH index buffer (MUST be called after `new MeshBVH(merged)`).
  Supports `MeshStandardMaterial`, `MeshPhysicalMaterial`, `MeshBasicMaterial`,
  material arrays (slot 0 + warn), and `ShaderMaterial` (white fallback + warn).
- `src/lib/utils/MaterialTextures.ts:29` — `buildMaterialTextures(perTri)`
  packs albedo + emissive into two square `ceil(sqrt(N))` RGBA Float DataTextures
  with NearestFilter, ClampToEdgeWrapping, no mipmaps.

### Lightmapper Bake Pass

**Direct light:**
- `src/lib/lightmap/LightmapperMaterial.ts` — `sampleAllLightsNEE` (≈line 288)
  iterates all lights in the DataTexture (up to `MAX_LIGHTS = 16`) via one
  shadow ray per light. `1/PI` dropped intentionally (D-006).
- Direct channel stores **raw irradiance** (hitAlbedo=vec3(1.0)), so material
  color is applied at composite time.

**Indirect / multi-bounce GI:**
- `src/lib/lightmap/LightmapperMaterial.ts` — `tracePath(...)` (≈line 315), a
  hand-unrolled loop bounded by `#define MAX_BOUNCES 4` and runtime `bounces`
  uniform. Emissive direct (term a), NEE on hit surface (term b), throughput
  multiply (term c), Russian Roulette from b≥2 (term d), next bounce hemisphere
  sample (term e).
- Sky-miss only on the first bounce (b==0): `if (b == 0) radiance += skyColor * skyIntensity`.

**Lights supported:** Point, Directional, Spot, Area (RectArea) via
`src/lib/lightmap/Lights.ts:56` `collectLightsFromScene`. All four light types
handled in `sampleLight` in `LightmapperMaterial.ts` (≈lines 235-278).

**Outputs:** MRT with 2 attachments — `directOut` (location 0) and `indirectOut`
(location 1) at `LightmapperMaterial.ts` ≈lines 145-146.

**AO** has been split to its own pass; no AO output from Lightmapper.

### AO Bake Pass

- `src/lib/lightmap/AOMaterial.ts` — standalone GLSL3 AO material. Single
  output `aoOut`. Stores RAW normalized visibility `t = clamp(dist/ambientDistance,
  0, 1)` per texel (1.0 on miss / hit beyond range).
- `src/lib/lightmap/AOMapper.ts:44` — `generateAOMapper(renderer, positions,
  normals, bvh, opts)`. Own `WebGLRenderTarget` (FloatType, LinearFilter),
  accumulator, render loop, dispose. Shape mirrors Lightmapper.
- `AOOptions.samples` is a live, independent ray budget (0–64 per frame).
  Default = `castsPerFrame` (LightmapBaker.ts:388).
- AO-only re-bake wired: `LightmapBakeResult.rebakeAO()` at
  `LightmapBaker.ts:309`; demo counterpart at
  `CornellBoxExample.ts:1667` (`rebakeAO()` private method).

imported GLBs). `TexelDensityMaterial.ts:33-35`.

### 4.6 — Per-mesh density multiplier fully wired in demo bin-packer

As noted in Section 3.2: `scaleInLightmap` → `perMeshScale` →
`binPackMeshes()` is live and fully working in the demo path
(`CornellBoxExample.ts:1091-1098`). Journal describes the field in S8 as a UI
control but leaves unclear whether it actually feeds the packer. It does.

### 4.7 — Auto-multi-atlas spawning when density target overflows one atlas

When the total UV demand of all meshes at the given `texelsPerMeter` cannot fit
one atlas, `binPackMeshes` automatically opens new atlas bins. Resulting
`numAtlases > 1` causes the demo to run N parallel bake groups. This is never
described as an automatic trigger in the journal — S8 describes it as a
"supported" feature but the mechanism (exceeding `fillRatio` → new bin) is not
documented.

### 4.8 — `generateAtlases` exported from `src/lib/index.ts`

`src/lib/index.ts:13` exports `generateAtlases`. The journal never mentions it
separately from `generateAtlas`. It is the multi-atlas serial wrapper required
by the demo.

### 4.9 — `AtlasViewer` multi-texture support (`setTextures`)

`src/lib/AtlasViewer.ts` exposes a `setTextures(texs: Texture[])` method used
at `CornellBoxExample.ts:1638`. The journal's S8 mentions "multi-atlas viewer"
in the commit summary but not that `AtlasViewer` has a dedicated multi-texture
API. Any migration must preserve this.

### 4.10 — `maybeBake` auto-bake gate (slider-release vs drag)

`CornellBoxExample.ts:1737` — `maybeBake(e?)` checks `e?.last === false` to
skip mid-drag events, and `!this.options.autoBake` to respect the manual mode.
This is the re-bake debounce mechanism. Not documented in any session.

### 4.11 — `fitCameraAndLightToScene()` on GLB import

`CornellBoxExample.ts:970` — auto-fits camera orbit target, distance, and
light position to the bounding box of any imported GLB scene. Not mentioned in
the S9 Task 12 reclassification.

### 4.12 — `mergeVertices` called on non-indexed GLB meshes at import

`CornellBoxExample.ts:944` — `if (!mesh.geometry.index) mesh.geometry = mergeVertices(mesh.geometry)`.
This converts non-indexed geometry to indexed, required by `extractPerTriangleMaterials`.
Not documented anywhere in the journal.

### 4.13 — `DilationMaterial` uses position G-buffer alpha as chart mask

`Refinement.ts:70` creates `DilationMaterial({ positions, resolution })` passing
the position texture. The shader reads `positions.a` to detect chart boundaries
(a=1 inside chart, 0 outside). This is the exact mechanism described in D-006
but never called out in any session as the implementation path.

### 4.14 — `pow(max(lit, 0), 1/1.1)` contrast boost in CompositeMaterial

`CompositeMaterial.ts:88` applies a subtle gamma-like contrast: `pow(max(lit, vec3(0.0)), vec3(1.0/1.1))`. This is live and affects every baked result. Not mentioned in any session log. It is not physically correct; it shifts the output's gamma curve. Any comparison of "before/after" that doesn't account for this will be misleading.

### 4.15 — `BakeErrorPhase` includes `'flood'` phase

`errors.ts:16` lists `'flood'` as a valid phase. No code currently throws
`BakeError` with phase `'flood'` — it is a reserved slot for the gap-flood
(dilation) step that is wired but the error tag is unused. Not mentioned in any
session.

## 5. Migration / Refactor Risk Register

The migration task is: "replace demo's own `bake()` orchestration with a call to
`LightmapBaker.bake()`." The following demo features exist TODAY in the demo
bake path but have NO equivalent in `LightmapBaker.bake()`:

### R-01 — `scaleInLightmap` per-mesh density multiplier
- **Demo path:** `CornellBoxExample.ts:1091`, `perMeshScale` dict → `binPackMeshes`.
- **Public API:** `perMesh` only has `{ resolution?, exclude? }` — no `scaleInLightmap`.
- **Risk:** Migration that deletes the demo's bin-pack code will silently lose
  per-mesh density weighting. The Tweakpane "Density ×" slider will become a no-op.

### R-02 — `binPackMeshes` multi-atlas auto-spawn logic
- **Demo path:** `CornellBoxExample.ts:1095-1102`, `numAtlases` drives the per-bin loop.
- **Public API:** `LightmapBaker.bake()` uses `partitionMeshes` which groups by
  `resolution`, not by density. Multi-atlas based on density is NOT in the public API.
- **Risk:** Migration loses density-aware atlas splitting. At high `texelsPerMeter`
  all meshes would go into one overloaded atlas.

### R-03 — Secondary directional light UI
- **Demo path:** `CornellBoxExample.ts:1154-1169`, `secondaryLightEnabled`,
  `secondaryDirX/Y/Z`, `secondaryIntensity`, `secondaryColor` build a `PackedLight`
  and append to the lights array.
- **Public API:** `LightmapBaker.bake()` calls `collectLightsFromScene()` which walks
  scene for `THREE.DirectionalLight` objects. If migrated, the secondary directional
  must be represented as a real `THREE.DirectionalLight` in the scene, not a
  UI-only virtual light.
- **Risk:** Migration could drop the secondary light if it isn't also added as a
  `THREE.DirectionalLight` to the scene.

### R-04 — `bakeBVH` retained for AO-only re-bake
- **Demo path:** `CornellBoxExample.ts:248`, `bakeBVH` stored; used in
  `rebakeAO()` at line 1668 to create new AOMapper without rebuilding BVH.
- **Public API:** `LightmapBakeResult.rebakeAO()` uses `this.internals.bvh`
  internally; this IS handled correctly. BUT the migration must ensure the demo
  calls `result.rebakeAO()` instead of its own `rebakeAO()`.
- **Risk:** If migration re-implements a new `rebakeAO` instead of delegating to
  `result.rebakeAO()`, the BVH retention logic must be duplicated correctly.

### R-05 — Multi-atlas progress widget (per-atlas sample line)
- **Demo path:** `CornellBoxExample.ts:1604-1617`, builds a per-atlas progress line
  showing `[done 256, 187, 187]` for up to 6 atlases.
- **Public API:** `BakeHooks.onProgress(phase, percent)` only provides a scalar.
  The per-atlas breakdown is lost.
- **Risk:** Migration loses granular multi-atlas progress display.

### R-06 — Per-mesh TexelDensityMaterial cache + per-mesh effective target
- **Demo path:** `CornellBoxExample.ts:1703`, `refreshTexelDensityMaterials()` builds
  one `TexelDensityMaterial` per mesh, with `target = globalTexelsPerMeter × perMesh.scaleInLightmap`.
- **Public API:** None. `TexelDensityMaterial` is exported as a primitive.
- **Risk:** Migration must replicate this per-mesh material cache or the "Texel Density"
  debug layer will lose per-mesh density scaling.

### R-07 — `autoApplyRefinement` post-bake refinement auto-trigger
- **Demo path:** `CornellBoxExample.ts:1576`, after bake done, if `autoApplyRefinement`
  is true, calls `applyRefinement()`.
- **Public API:** `LightmapBakerOptions.denoise = true` runs refinement inside
  `bake()` automatically. But the demo exposes this as a separate manual/auto UI
  option with its own "Run Refinement" button that can be re-run after the fact.
  The "Revert to Raw" button and `showUnrefined()` method have no equivalent.
- **Risk:** Migration loses the ability to re-run refinement with new parameters
  without re-baking.

### R-08 — `atlasViewer.setTextures(texs)` multi-atlas viewer
- **Demo path:** `CornellBoxExample.ts:1638`. Shows ALL groups' textures simultaneously.
- **Public API:** `AtlasViewer` is exported; `setTextures` is a method on it.
- **Risk:** Migration must replicate the "collect textures from all groups for the
  active layer" logic.

### R-09 — GLB camera/light auto-fit on import
- **Demo path:** `CornellBoxExample.ts:970` `fitCameraAndLightToScene()`.
- **Public API:** None.
- **Risk:** A migration that uses `LightmapBaker.bake()` still needs to auto-fit
  camera and light for imported GLBs — this is a demo-specific feature the public
  API will never own.
